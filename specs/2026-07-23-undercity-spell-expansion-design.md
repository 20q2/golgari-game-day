# The Undercity — Spell Expansion & Single-Source Refactor (design)

**Date:** 2026-07-23
**Status:** Approved design, pending implementation plan.
**Related:** [undercity-spells.md](undercity-spells.md) (living reference), [2026-07-10-undercity-spells-design.md](2026-07-10-undercity-spells-design.md) (original rationale).

## Goal

Two things, in one pass:

1. **Add ~13 new spells** across tiers, reusing the existing effect-kind
   vocabulary (no new dispatcher branches), plus 6 new one-battle buff/curse
   kinds for build diversity.
2. **Kill the Python↔TS spell duplication** so a spell is authored once. The
   Python tables stay the source of truth; the client mirror becomes generated
   and drift-guarded — mirroring the established `map.json` → `sync_map.py`
   pattern.

Non-goals: no new *effect* kinds (AoE, DoT, dispel, summon, shields); no phase-3
book-acquisition work; no change to the cast flow, cooldown model, or the
"spells never kill" invariant. Balance numbers here are first-pass and tunable
later via the `tune-undercity-balance` skill.

## Part 1 — Single-source architecture (codegen)

### Source of truth

`infrastructure/lambda/undercity_data.py` keeps `SPELLS`, `GRIMOIRES`, and
`BIOME_SPELLS` as the authored source. Each `SPELLS` entry gains two
client-facing fields the server ignores at runtime:

- `icon` — Material Icons ligature name (was previously only in `spells.ts`).
- `desc` — the cast-picker description (was previously only in `spells.ts`).

`blurb` stays: it remains the server's cast-toast flavor text (`_resolve_spell_effect`
uses it in `self_buff` result text). `desc` is the client picker line; `blurb`
is the in-game toast. They are allowed to differ.

Precedent for carrying client-only fields in a server-owned data source:
`map.json` already carries client-only `regions`/`decals`.

### Generator

New script `infrastructure/lambda/sync_spells.py` (sibling to `sync_map.py`):

- Imports `undercity_data`.
- Emits `src/app/undercity/data/spells.generated.ts` containing **only** the
  three data arrays (`SPELLS`, `GRIMOIRES`, `BIOME_SPELLS`) plus a
  `// AUTO-GENERATED by sync_spells.py — DO NOT EDIT` header.
- Field order and formatting are deterministic (stable output → clean diffs,
  reliable staleness check).

The generated TS must type-check against the hand-written types, so the
generator emits values matching `SpellInfo` / `GrimoireInfo` (e.g. `tier` as a
numeric literal, `spells: string[]`).

### `spells.ts` becomes logic-only

`src/app/undercity/data/spells.ts` keeps everything that is *not* raw data and
re-exports the generated arrays:

- Types: `SpellEffect`, `SpellInfo`, `GrimoireInfo`.
- Re-exports `SPELLS`, `GRIMOIRES`, `BIOME_SPELLS` from `spells.generated`.
- Derived: `SPELL_MAP`, `GRIMOIRE_MAP`.
- Helpers: `spellPower`, `spellPowerLabel`, `cooldownLeftMin`,
  `grimoireSwapLeftMin`, `spellCategoryStyle`.
- Unrelated mirrors that currently live here (`GRIMOIRE_CAPACITY`,
  `WITCH_SCROLL_STOCK`, `GRIMOIRE_SWAP_COOLDOWN_MIN`, `SPELL_POWER_PER_LEVEL`)
  stay hand-written — they are config mirrors, not the spell tables.

No consumer of `spells.ts` changes. No async. Static imports and helper
signatures are preserved.

### Drift guard

New pytest in `infrastructure/lambda/tests/` (mirrors `test_map_file.py`):
regenerates the TS in memory and asserts it byte-matches the committed
`spells.generated.ts`. The suite fails while they differ, exactly like the map
copies-match test.

### Going-forward authoring flow

Edit the Python table → run `python infrastructure/lambda/sync_spells.py` →
commit both files. `npm run build` + `pytest` are the gates.

## Part 2 — The new spells

All reuse existing effect kinds. Base values shown; damage/heal/boss magnitudes
level-scale at runtime via `engine.spell_power` (`SPELL_POWER_PER_LEVEL = 1.0`).
Buff/curse/traversal values are flat.

|---|---|---|---|---|---|---|---|

| `necrotic_lance` | Necrotic Lance | 2 | field_damage | — | 16 | 9 | 28 |
| `withering_gout` | Withering Gout | 3 | field_damage | — | 26 | 6 | 26 |

damage → `flash_on`/`bolt`, heal → `healing`, DEF buff → `shield`,
SPD → `directions_run`, ATK buff → `local_fire_department`, curse →
`heart_broken`.

### New buff kinds (engine wiring)

Six new `buffKind`s. Each is:

1. One `elif kind == '<kind>':` branch in `engine.effective_stats()`
   (`undercity_engine.py`, alongside `rot_surge`/`harden_shell`/…). Self-buffs
   apply `* mult` (Squirrel Warrior doubling); curses use `max(1, eff[stat] - n)`
   and ignore `mult`, matching the existing `bone_chill`/`weaken_hex` pattern.
2. One entry appended to `ONE_BATTLE_BUFFS` (`undercity_db.py`) so it is
   consumed when the next battle ends.

| buffKind | effective_stats effect |
|---|---|
| `savage_roar` | `eff['atk'] += 5 * mult` |
| `iron_hide` | `eff['def'] += 4 * mult` |
| `fleetfoot` | `eff['spd'] += 3 * mult` |
| `warding_dance` | `eff['def'] += 3 * mult; eff['spd'] += 3 * mult` |
| `sap_vigor` | `eff['spd'] = max(1, eff['spd'] - 3)` (curse) |
| `rust_curse` | `eff['def'] = max(1, eff['def'] - 4)` (curse) |

No flee-affecting buffs (would need `_combatant` changes) — out of scope.

### New grimoires

Bundle the new spells and give them delivery. Tier-I books are shop-buyable now;
tier-II/III books are defined for the data model and reachable via the shipped
scroll system (below) until phase-3 book acquisition ships. Proposed bundles
(exact costs finalized in the plan, consistent with existing tier pricing):

| id | Name | Tier | Cost | Spells |
|---|---|---|---|---|
| `skirmishers_notes` | Skirmisher's Notes | 1 | 32 | `ember_fleck` |
| `bulwark_breviary` | Bulwark Breviary | 2 | 70 | `iron_hide`, `renewing_bloom` |
| `snipers_folio` | Sniper's Folio | 2 | 70 | `necrotic_lance`, `fleetfoot_draught` |
| `saboteurs_libram` | Saboteur's Libram | 2 | 70 | `sap_vigor`, `shadowstep` |
| `berserkers_roll` | Berserker's Roll | 2 | 72 | `savage_roar`, `ember_fleck` |
| `throneburner_codex` | Throneburner Codex | 3 | 150 | `sear_throne`, `withering_gout`, `rust_curse` |
| `warding_tome` | Warding Tome | 3 | 150 | `warding_dance`, `deep_mend` |

Convention (matches all existing books): every book's member spells are at or
below the book's own tier. `skirmishers_notes` is the one shop-buyable tier-I
book of this batch (immediate content for the bazaar). The tier-II/III books are
defined for the data model and reached via scrolls until phase-3 book
acquisition ships.

### How players get the new spells today

The shipped **scroll system** (Sedgemoor Witch, commit series ending `11f6662`)
distributes spells by tier from reward sources (`SCROLL_DROP_TIER`:
loot/mystery → T1, elite/dig/cache → T2, higher → T3) and lets players inscribe
scrolls into grimoires. So every new spell is reachable immediately via scrolls
+ the witch, independent of unbuilt phase-3 book drops. `WITCH_SCROLL_STOCK`
(the witch's fixed T1 scroll stock) may optionally gain a new T1 spell; treated
as a balance tweak, not required.

## Part 3 — Tests & docs

### Tests (`infrastructure/lambda/tests/`)

- **Existing data-integrity tests** already enforce required fields per effect
  kind (`test_undercity_spells.py`), so the new spell entries are auto-validated
  once added.
- **New buff-kind tests:** for each of the 6 kinds, assert the stat delta shows
  up in `engine.effective_stats()` and that the buff is cleared by
  `_consume_one_battle_buffs` after a battle (self-buffs) / applied to the target
  then consumed (curses). Follow the existing `self_buff` / `field_curse` test
  shapes.
- **Codegen guard:** `test_spells_generated_in_sync` regenerates in memory and
  compares to the committed `spells.generated.ts` (mirrors `test_map_file.py`).
- Full suite (`python -m pytest tests -q`) stays green.

### Client build

`npm run build` must stay green after regenerating `spells.generated.ts`.

## Invariants preserved

- Spells never kill (all damage/boss paths already floor at 1; new spells add no
  new damage path).
- Cooldowns only start on successful cast (unchanged dispatch).
- Grimoire collection is permanent; loadouts are fixed bundles (new books are
  fixed bundles like the rest).
- Python is the single authored source; the client mirror is generated and
  guarded.

## Deploy note

Backend changes (`undercity_data.py`, `undercity_engine.py`, `undercity_db.py`)
require a `cdk deploy` before the live client can cast the new spells. Per
project convention the user runs deploys; implementation ends with tests +
build green and a deploy note.
